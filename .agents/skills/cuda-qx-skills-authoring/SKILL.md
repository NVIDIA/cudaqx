# Authoring CUDA-QX Agent Skills

The meta skill: how the `.agents/skills/` tree works and how to
extend it. Most users will never need this; you only land here if
you're editing skill content, the sync script, or the eval harness.

## Inputs

Caller provides:

- Skill intent: add a new skill / edit an existing one / split or
  merge / onboard a new target agent.
- For new skill: name, domain, audience, workflows it covers.
- For edits: which skill, what changes, why.
- For onboarding an agent: target tool name + discovery path convention.

## Outputs

This skill produces:

- New or updated `.agents/skills/<name>/SKILL.md` + `references/`
  (with full frontmatter, Inputs/Outputs contract, repo tag).
- Updated `AGENTS.md` table.
- Synced `.claude/skills/` and `.cursor/skills/` mirrors.
- A passing `sync_agents_skills.sh --check`.

Does NOT produce: changes to skill *content* outside its own concern
(skill authors edit one skill at a time); changes to actual code under
`libs/` (→ domain skills).

## Audience

Maintainers and skill contributors. Familiarity with Markdown +
YAML frontmatter, plus enough bash to read `sync_agents_skills.sh`.

## First three actions

```bash
ls .agents/skills
bash scripts/sync_agents_skills.sh --check
cat AGENTS.md | head -50
```

The check confirms the mirror is in sync before you start; if it's
already stale, fix that first.

| Area | Path |

| AGENTS.md | repository root |
| Eval harness | `.agents/evals/` (`README.md`, `prompts/`, `graders/`, `assertions/`, `runners/`) |

## The canonical / mirror model

Canonical: `.agents/skills/`. Edit *only* here.

Mirrors: `.claude/skills/`, `.cursor/skills/`. Generated by
`scripts/sync_agents_skills.sh` via `cp -r`. **No path rewriting** —
the mirror is byte-identical to the canonical, plus a `.GENERATED`
sentinel file. Both mirrors are gitignored.

Implication: paths inside a SKILL.md (e.g.
`bash .agents/skills/_shared/scripts/preflight.sh`) appear *as-is*
in every mirror. This is intentional — agents see the canonical
path and run the script from there. Do not hard-code
`.claude/skills/` or `.cursor/skills/` in skill content.

CI gate:

```bash
bash scripts/sync_agents_skills.sh --check
```

Run this before committing skill edits.

## Frontmatter contract

Every SKILL.md begins with a YAML block. Required fields:

```yaml
---
name: "kebab-case-name"
title: "Human Title"
description: >-
  One paragraph. Should be specific enough that a routing model can
  decide whether to invoke this skill from the description alone.
  Include trigger keywords (decoder names, function names,
  vocabulary the user will type).
version: "0.2.1"
author: "CUDA-QX"
license: "Apache License 2.0"        # or LicenseRef-NVIDIA-Proprietary for QEC-restricted
compatibility: "..."                    # platforms, Python versions
tags: [list, of, tags]
tools: [Read, Glob, Grep, Bash]        # tools the skill expects to use
metadata:
  author: "CUDA-QX"
  domain: "..."
  audience: [list, of, personas]
  languages: [python, c++, ...]
---
```

The `description` is what routing logic sees. Spend time on it. Two
heuristics:

1. **Trigger phrases first.** "Use when the user mentions X, Y, Z."
2. **Negative space.** "Do NOT use this for X — that's in skill Y."

## Conventions

1. **One responsibility per skill.** A skill covers one workflow,
   one audience, one set of decisions. Splitting `qec` into
   `qec-realtime`, `qec-ai-decoders`, `qec-extending`, etc., is
   the right move once a skill exceeds ~500 lines of prose.

2. **Skill body structure**: Description → How to read this skill
   → Audience → First three actions → Key Paths → Workflow Index →
   Conventions → Quick Start → Self-Check Protocol → When stuck →
   Additional resources. Mimic existing skills.

3. **Concrete file paths everywhere.** Names like `nv-qldpc-decoder`
   or "the YAML config" without a path are useless to an agent.
   Always say "see `libs/qec/include/cudaq/qec/decoder.h`".

4. **No invented APIs.** Grep the source before claiming a function
   exists. Skills hallucinate when authors do.

5. **References are companion files**, not separate skills. Keep
   them inside `references/` of the parent skill. Cross-reference
   liberally between related skills.

6. **Tone**: short sentences, concrete, no marketing. The audience
   is an agent or a developer; both prefer signal over fluff.

7. **No source paths inside `_shared/`** that aren't really shared.
   `_shared/` is for files used by ≥2 skills. Skill-specific helpers
   live inside the skill directory.

8. **Re-run sync after every edit.** Otherwise mirrors drift and
   CI catches it.

## Quick start: add a new skill

```bash
mkdir -p .agents/skills/cuda-qx-NEW/references
$EDITOR  .agents/skills/cuda-qx-NEW/SKILL.md
$EDITOR  .agents/skills/cuda-qx-NEW/references/<topic>.md
bash scripts/sync_agents_skills.sh
# Update the table in AGENTS.md if appropriate
git add .agents/skills/cuda-qx-NEW AGENTS.md
git commit -s -m "skill: add cuda-qx-NEW"
```

Then:

1. Test the skill from a Claude Code session: see if it gets routed
   when you type the trigger phrases.
2. Add an eval (`references/evals.md`) if the skill covers a
   workflow worth measuring.

## Self-Check Protocol

```
[ ] SKILL.md has full frontmatter; `description` includes trigger
    phrases and a "do NOT use" line where relevant.
[ ] At least 2 references under references/ (skills with only a
    SKILL.md are usually under-developed).
[ ] All paths in the body are real files (grep to verify).
[ ] sync_agents_skills.sh --check passes.
[ ] AGENTS.md updated if this is a new top-level skill.
[ ] Cross-references to related skills are present.
[ ] No hard-coded `.claude/skills/` or `.cursor/skills/` paths in
    canonical content.
[ ] If the skill needs preflight, it uses the shared scripts under
    `_shared/scripts/`.
```

## When stuck

1. Re-read an existing skill (`cuda-qx-qec-decode/SKILL.md` is the most
   developed) and copy structure.
2. Run `sync_agents_skills.sh` (no `--check`) and inspect
   `.claude/skills/` to confirm content arrives intact.
3. For evals: read `.agents/evals/README.md`.
4. For sync issues: read `scripts/sync_agents_skills.sh` end-to-end
   (~180 lines).
