BRANCH: decoding-server-parallel-decoders (forked from decoding-server-udp-transport) — updated 2026-07-06

WHAT THIS BRANCH CONTAINS: the decoding-server architecture from PR #653 (Vedika's DecoderServer: one DecoderSession worker thread per configured decoder, SessionRegistry, RpcDispatcher, ITransceiver transport abstraction, RoundAccumulator + syndrome mapping table, CqrTransceiver bridging cudaq-realtime DeviceCallService, LoopbackTransceiver for tests), merged with this branch's spec-compliant wire format, concurrency evidence, daemon integration, and two-process test coverage. Decoder execution is concurrent across decoders by construction: each DecoderSession owns a FIFO worker thread, so N logical qubits (N decoder ids) decode in parallel.

ARCHITECTURE (libs/qec/lib/realtime/decoding-server-cqr/):
- DecoderServer: owns the SessionRegistry + RpcDispatcher, runs one blocking receive loop per unique transceiver; constructible from a YAML path, a parsed multi_decoder_config, or explicit transceivers (LoopbackTransceiver for tests).
- DecoderSession: owns one decoder instance + a dedicated FIFO worker thread + a bounded work queue (depth 64). enqueue_syndromes is fire-and-forget (queue full -> the drop is latched and reported as SYNDROMES_DROPPED at the next get_corrections); get_corrections is a rendezvous through the queue (per-decoder FIFO guarantees all prior enqueues executed first); a decoder exception during a deferred enqueue is latched STICKY and surfaced as INTERNAL_ERROR at the next get_corrections (reset clears it). Per-session metrics + a global busy high-water mark.
- RoundAccumulator + SyndromeMappingTable: the spec's syndrome mapping table (per-VP round-relative scatter, multi-fragment round assembly). The MVP default is a single-VP identity row (empty index list = the fragment passes through as the round at whatever length arrives; the decoder accumulates rounds itself).
- CqrTransceiver: bridges cudaq-realtime host-call handlers to the server. Response-bearing RPCs (get_corrections / reset_decoder) block the handler until the session worker sends the response; enqueue_syndromes is ACKed immediately (ACCEPTED) so the single transport dispatcher thread never parks on decoder execution — that preserves cross-decoder parallelism on the wire.
- decoding_server_cqr.cpp: the DeviceCallService plugin. Spec-schema entries (4-arg enqueue, 3-arg get_corrections, 1-arg reset). Config comes from CUDAQ_QEC_DECODER_CONFIG (daemon path, decoders built eagerly before the READY line) or from the last configure_decoders() call in-process (application path, server built lazily on the first RPC). Handler-level exception guards return INTERNAL_ERROR instead of crashing the dispatcher. --save_syndrome capture preserved. atexit + explicit shutdown hooks join the server thread cleanly.

WIRE FORMAT: decoder_server_runtime.md exactly (lowering-accurate, no padding) — enqueue payload is decoder_id, counter, syndrome_mapping_id, then syndrome_bits as a CUDAQ_TYPE_BIT_PACKED vector<bool>: a u64 element-count prefix (== num_syndromes) followed by ceil(N/8) LSB-first bit-packed bytes, so arg_len = 32 + B with NO trailing pad (the redundant explicit num_syndromes scalar was dropped — it is the vector's own prefix). get_corrections carries decoder_id + return_size (the OUT vector<bool> length) + reset (trailing 1-byte bool), arg_len = 17, NO padding, and its response returns ceil(R/8) bit-packed correction bytes with result_len = ceil(R/8) (no pad). reset_decoder is decoder_id only (arg_len = 8). RPCHeader/RPCResponse framing verbatim. Matches cudaq-spec proposals/decoder_server_runtime.md @ 725b33e.

DAEMON (tools/decoding-server/qec_decoding_daemon): validates the YAML, exports it via CUDAQ_QEC_DECODER_CONFIG, and lets the DecoderServer own the decoder instances (no double instantiation). Prints QEC_DECODING_DAEMON_READY / DISPATCHED / MAX_CONCURRENT_DECODERS (high-water mark of simultaneously-busy session workers).

PREREQUISITE: the paired CUDA-Q branch decoding-server-udp-transport (in /workspaces/cudaq) installed at /usr/local/cudaq — it provides the udp / cpu_roce device_call channels. Install order matters: 1) cmake --build /workspaces/cudaq/realtime/build --target install, then 2) reconfigure + cmake --build /workspaces/cudaq/build --target install (configure log must say "device_call: udp channel enabled").

BUILD:
  cmake -S . -B build -DCMAKE_BUILD_TYPE=Release -DCUDAQ_DIR=/usr/local/cudaq/lib/cmake/cudaq
  cmake --build build

RUN THE TESTS:
  cd build && ctest -R "DecodingServerTwoProcess"
    -> TwoProcessHostDispatch, TwoProcessHostDispatchMultiErrorLut, TwoProcessHostDispatchDualDecoders (one __qpu__ kernel driving TWO decoders in one daemon: per-decoder correctness, no cross-talk, dispatch and concurrency-watermark assertions)
  cd build && ctest -R "realtime|surface_code|pymatching_device"
    -> surface-code app_examples including surface_code-1-cqr-two-process (full memory experiment decoded by a separate daemon) and the in-process host_dispatch variants
  NOTE: the Python variants (app_examples.py-*) need PYTHONPATH=/usr/local/cudaq:$PWD/python (from the build dir).
  Transport for the two-process tests (DecodingServerTwoProcess and surface_code-1-cqr-two-process) is switchable via QEC_DECODING_SERVER_TRANSPORT=cpu_roce (RDMA topology from the CUDAQ_CPU_ROCE_TEST_* env vars; works over SoftRoCE/rdma_rxe, e.g. a self-loop rxe device on a veth pair); default udp loopback runs anywhere. NOTE: the cpu_roce ring geometry is part of the wire contract -- the caller channels pin --cudaq-device-call-slots=8 / --cudaq-device-call-slot-size=256 to match the daemon's --num-slots/--slot-size defaults.

RUN THE SERVER MANUALLY (surface-code demo, two processes):
  1) generate config: build/libs/qec/unittests/realtime/app_examples/surface_code-1-cqr --distance 5 --num_rounds 5 --num_shots 1000 --save_dem config.yml --decoder_type multi_error_lut
  2) start daemon:    build/bin/qec_decoding_daemon --config=config.yml --transport=udp --port=0   (read the port from QEC_DECODING_DAEMON_READY; shutdown prints DISPATCHED + MAX_CONCURRENT_DECODERS)
  3) run the client:  QEC_DECODING_SERVER_PORT=<P> build/libs/qec/unittests/realtime/app_examples/surface_code-1-cqr --distance 5 --num_rounds 5 --num_shots 1000 --load_dem config.yml
  For N logical qubits, put N decoder entries (ids 0..N-1) in the YAML — each gets its own DecoderSession worker and their decodes run concurrently.

CONFIG NOTES: decoder entries accept an optional per-decoder `transport: cpu_roce|gpu_roce` field (from PR #653) for the future standalone ITransceiver deployment; all entries in one server must currently agree. The daemon's --transport=udp|cpu_roce flag selects the cudaq-realtime channel and is independent of that field.
