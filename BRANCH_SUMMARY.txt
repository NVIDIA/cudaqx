BRANCH: decoding-server-parallel-decoders (forked from decoding-server-udp-transport) — 2026-07-06

WHAT THIS BRANCH ADDS: per-decoder execution workers in the realtime decoding server. Previously all decoders executed inline on the single transport dispatcher thread, so with multiple decoders configured (one per logical qubit) every decoder's work serialized behind every other's. Now each configured decoder owns a worker thread and a bounded FIFO ring of request slots, so N logical qubits decode concurrently.

CHANGES (6 files):
1. libs/qec/lib/realtime/realtime_decoding.cpp / .h — the worker infrastructure. Each decoder gets a worker thread + an 8192-slot request ring at configure time (always on, no config knob). The dispatcher thread VALIDATES and STAGES a request — writing the syndrome payload directly into the decoder's ring slot — and returns immediately for the fire-and-forget calls (enqueue_syndromes, reset_decoder); decoder execution happens on the decoder's own thread. get_corrections stages a rendezvous and blocks until that decoder's queue drains (end-of-shot only, bounded by one decode tail). Semantics: per-decoder FIFO order is preserved (single staging producer + FIFO ring); cross-decoder ordering is deliberately given up — that is the parallelism. Validation errors still throw synchronously in the caller; a decoder exception during deferred execution becomes STICKY per-decoder state, rethrown at that decoder's next get_corrections (reset_decoder clears it). Ring full -> staging blocks (natural backpressure onto the transport). New host APIs: stage_syndromes / commit_syndromes (zero-copy staging for transport handlers) and max_concurrent_decoder_workers() (busy high-water mark).
2. libs/qec/lib/realtime/decoding-server-cqr/decoding_server_cqr.cpp — the daemon's enqueue_syndromes handler now unpacks the wire's LSB-first bits DIRECTLY into the decoder's ring slot via stage_syndromes/commit_syndromes (the old path did two copies: unpack to a stack buffer, then copy into the decoder). "Success" on the wire means ACCEPTED; a deferred decoder error surfaces at that decoder's next get_corrections.
3. libs/qec/tools/decoding-server/qec_decoding_daemon.cpp — prints QEC_DECODING_DAEMON_MAX_CONCURRENT_DECODERS count=<N> at shutdown (next to the existing DISPATCHED line): the high-water mark of simultaneously-busy decoder workers, i.e. concurrency evidence.
4. libs/qec/unittests/realtime/test_decoding_server.cpp — DaemonProcess test helper improvements (absolute --config paths, configurable READY timeout, captures the MAX_CONCURRENT line) and a new test, DecodingServerUdp.TwoProcessHostDispatchDualDecoders: one __qpu__ kernel drives TWO pymatching decoders (two logical qubits) in one daemon over UDP — asserts each decoder returns the correction for its own syndrome, no cross-talk between decoders, >= 6 dispatched calls, and worker high-water mark >= 1.
5. cmake/Modules/CUDA-QX.cmake — comment only: documents that cudaqx_add_device_code's nvq++ custom command has no header dependencies (no depfile), so header changes do NOT trigger recompilation; rm the stale <target>_<source>.o under the build tree after header changes until depfile output is wired up.

PREREQUISITE: the paired CUDA-Q branch decoding-server-udp-transport (in /workspaces/cudaq) installed at /usr/local/cudaq — it provides the udp / cpu_roce device_call channels. Install order matters: 1) cmake --build /workspaces/cudaq/realtime/build --target install (the realtime package exports the udp-transport target), then 2) reconfigure + cmake --build /workspaces/cudaq/build --target install (the configure log must say "device_call: udp channel enabled").

BUILD:
  cmake -S . -B build -DCMAKE_BUILD_TYPE=Release -DCUDAQ_DIR=/usr/local/cudaq/lib/cmake/cudaq
  cmake --build build

RUN THE TESTS:
  cd build && ctest -R "DecodingServerUdp"
    -> TwoProcessHostDispatch, TwoProcessHostDispatchMultiErrorLut, TwoProcessHostDispatchDualDecoders (each spawns its own qec_decoding_daemon on an ephemeral UDP port; the dual test is the multi-decoder concurrency coverage)
  cd build && ctest -R "realtime|surface_code|pymatching_device"
    -> the surface-code app_examples, including surface_code-1-cqr-two-process (full surface-code memory experiment decoded by a separate daemon process over UDP)
  NOTE: the Python variants (app_examples.py-*) need PYTHONPATH=/usr/local/cudaq:$PWD/python (from the build dir) or they fail collecting with "module 'cudaq' has no attribute 'measure_handle'".
  Transport is switchable for the gtests via QEC_DECODING_SERVER_TRANSPORT=cpu_roce (uses the CUDAQ_CPU_ROCE_TEST_* env vars for RDMA topology); default is udp loopback which runs anywhere.

RUN THE SERVER MANUALLY (surface-code demo, two processes):
  1) generate config: build/libs/qec/unittests/realtime/app_examples/surface_code-1-cqr --distance 5 --num_rounds 5 --num_shots 1000 --save_dem config.yml --decoder_type multi_error_lut
  2) start daemon:    build/bin/qec_decoding_daemon --config=config.yml --transport=udp --port=0   (reads back "QEC_DECODING_DAEMON_READY port=<P>"; at shutdown prints the DISPATCHED and MAX_CONCURRENT_DECODERS counts)
  3) run the client:  QEC_DECODING_SERVER_PORT=<P> build/libs/qec/unittests/realtime/app_examples/surface_code-1-cqr --distance 5 --num_rounds 5 --num_shots 1000 --load_dem config.yml
  For N logical qubits, put N decoder entries (ids 0..N-1) in the YAML — each gets its own execution worker and their decodes run concurrently; the sample dual-decoder YAML shape is in the TwoProcessHostDispatchDualDecoders test.
